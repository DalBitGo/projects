# 프로젝트 목표 및 구성 요소별 존재 이유

이 문서는 "Project Inspector"의 최종 목표는 무엇이며, 프로젝트를 구성하는 각 폴더, 파일, 주요 함수/클래스가 그 목표를 달성하기 위해 궁극적으로 어떤 역할을 하는지 설명합니다.

---

### 🎯 프로젝트 전체의 최종 목표

- **"개발자가 특정 코드 프로젝트를 손쉽게 분석하고 이해할 수 있도록 돕는다."
**    이것이 이 "Project Inspector"의 단 하나의 최종 목표입니다. 아래의 모든 파일과 코드는 이 목표를 달성하기 위한 부품일 뿐입니다.

---

### 📂 파일 및 폴더별 존재 이유

#### 📁 `frontend` 폴더
- **존재 이유**: 사용자가 복잡한 명령어 없이도, **친숙한 웹 브라우저 화면을 통해** 모든 분석 기능을 클릭 몇 번으로 사용할 수 있게 하기 위함입니다.
- **`+page.svelte`**: 이 화면을 실제로 그리는 설계도입니다. "경로 입력창", "실행 버튼" 등을 만들어, 사용자의 요청을 백엔드로 전달하는 **'창구'** 역할을 하기 위해 존재합니다.

#### 📁 `backend` 폴더
- **존재 이유**: 실제 분석을 수행하는 **'엔진'** 역할을 하기 위함입니다. 눈에 보이지 않는 곳에서 프론트엔드가 요청한 실제 작업들을 처리합니다.
- **`main.py`**: 백엔드의 **'총괄 매니저'**입니다. 프론트엔드로부터 들어온 요청("파이썬 코드 합쳐줘!")을 접수하고, 어떤 직원에게 그 일을 시킬지 결정(`TASK_DISPATCHER`를 통해)한 뒤, 작업이 끝나면 결과를 다시 프론트엔드에 전달하기 위해 존재합니다.
- **`utils` 폴더**: 실제 '일'을 하는 **'전문 작업자'**들을 모아놓은 곳입니다.
    - **`code_merger.py`**: "여러 파일에 흩어져 있는 코드를 하나로 합쳐달라"는 요청을 전문적으로 처리하기 위해 존재합니다.
    - **`structure_analyzer.py`**: "파이썬 코드의 설계도(클래스, 함수 구조)를 그려달라"는 가장 어려운 요청을 전문적으로 처리하기 위해 존재합니다.
    - **`constants.py`**: 작업자들이 공통적으로 사용하는 규칙("이 폴더들은 분석하지 마!")을 한 곳에 모아두어, 규칙이 바뀌었을 때 여러 곳을 수정할 필요가 없게 만들기 위해 존재합니다.

---

### 🧩 주요 함수/클래스별 존재 이유

#### `structure_analyzer.py` 내부
- **`analyze_structure` 함수**: "구조 분석"이라는 작업의 **'총 책임자'**입니다. 전체 폴더를 분석할지, 파일 하나만 분석할지 결정하고, 실제 분석 작업자(`analyze_python_file`)에게 일을 분배하기 위해 존재합니다.
- **`analyze_python_file` 함수**: 파이썬 코드를 받아, `ast`라는 특수 도구를 사용해 코드를 **'해부'**하는 역할을 합니다. 이 함수가 있기에 코드를 단순 텍스트가 아닌, 클래스, 함수 등의 구조로 이해할 수 있습니다.
- **`CallVisitor` 클래스**: '해부'된 코드 내에서, "A 함수가 B 함수를 부른다" 와 같은 **'관계'**를 찾아내는 **'탐정'** 역할을 합니다. 이 클래스 덕분에 코드의 의존성을 파악하는 심층 분석이 가능해집니다.

#### `main.py` 내부
- **`process_project` 함수**: 프론트엔드에서 온 요청을 처리하는 **'실무 총괄'**입니다. "경로에 문제는 없는지?", "요청한 작업이 등록된 작업인지?" 등을 확인하고, 실제 작업자(utils 함수)에게 일을 넘긴 뒤, 문제가 생기면 에러를 보고하는 역할입니다.
- **`TASK_DISPATCHER` 딕셔너리**: "파이썬 합치기 작업은 `code_merger`에게, 구조 분석은 `structure_analyzer`에게" 와 같이, 요청 종류에 따른 **'담당자 목록'**을 만들어 둔 것입니다. 이 목록 덕분에 새로운 작업이 생겨도 담당자만 추가하면 되므로 일이 편해집니다.
